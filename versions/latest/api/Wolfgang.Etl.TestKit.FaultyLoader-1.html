An in-memory loader for testing error-handling paths. It consumes the source stream like a normal loader, but can be configured to inject deterministic faults — throwing at a given item, throwing after completion, or duplicating an item — so consumers can exercise mid-stream failure, finalization failure, and idempotency logic without hand-rolling broken fakes.

Remarks

When constructed with collectItems: true, every loaded item — including any duplicates — is accumulated and exposed via GetCollectedItems().

Faults are configured through the fluent ThrowAt(int, Exception), ThrowAfterCompletion(Exception), and DuplicateAt(int) methods, each of which returns the same instance so calls can be chained. Multiple faults stack on a single instance.

Fault indices are zero-based and refer to the position in the loaded (post-skip) sequence. A configured fault fires after Wolfgang.Etl.Abstractions.LoaderBase<TDestination, TProgress>.IncrementCurrentItemCount() for that item, so a progress report reflects the item that caused the failure. When a ThrowAt(int, Exception) and a DuplicateAt(int) are configured for the same index, the throw takes precedence and the duplicate is not loaded. Calling ThrowAt(int, Exception) twice for the same index replaces the earlier exception (last-wins).

ThrowAt(int, Exception)

Configures the loader to throw exception when it reaches the item at the specified zero-based index in the loaded sequence. The failing item is counted (its Wolfgang.Etl.Abstractions.LoaderBase<TDestination, TProgress>.IncrementCurrentItemCount() runs) before the exception is thrown, so progress reflects the item that caused the failure, but the item itself is not stored.
